Section 2

Executive Summary

This report looks at a large subset of NYC 311 Service Requests, which include non-emergency complaints submitted by residents across New York City. It captures information like complaint types, submission and closure dates, locations, boroughs, responsible agencies, and resolution details.

We analyzed 5 million records with 12 variables, covering several years of reporting and reflecting how residents typically submit complaints in a big city.

The dataset is rich and large enough for exploration, visualization, and basic statistical modeling. Some data quality issues were noted, such as missing closure dates, incomplete location info, and inconsistencies in complaint types or borough names.

Duplicates were mostly handled during SQL ingestion, though some logical duplicates exist when multiple complaints describe the same issue. Temporal fields needed careful parsing to ensure accurate response-time analysis.

Section 3

Dataset Overview

The dataset is a sampled portion of NYC 311 Service Requests from NYC Open Data. It includes 5 million rows and 12 columns, representing a typical large public sector dataset.

Originally in CSV format, the data was loaded into SQLite for structured querying and preprocessing. All analysis used a single table (nyc311_raw) with no joins required.

The Unique Key column serves as a logical identifier for each complaint. There are no explicit foreign keys.

Each row represents a single 311 complaint, including submission details, status, location, and resolution. The unit of observation is an individual service request.

Section 4

Schema and Metadata Summary

The dataset includes dates, categories, text, numbers, and location info.

Key points about the schema:

  • Each complaint has a unique identifier
  • Created and closed timestamps
  • Complaint type and descriptor fields
  • Status, borough, and agency name
  • Location: ZIP code, latitude, longitude
  • Free-text resolution description

Data types were inferred from the CSV, then validated and cleaned in Python. Dates were parsed, numbers coerced where necessary, and categories standardized. Some columns can be null, especially those related to closures or geographic info.

Expected values include valid NYC borough names, coordinates within NYC bounds, and created dates that come before closed dates.

Section 5

Data Types and Structural Validation

Initial checks found some common real-world inconsistencies.

Dates were stored as strings and had to be converted. Some entries were missing or malformed and were set to null.

Latitude, longitude, and ZIP codes had mixed formats and missing values. ZIP codes were treated as categorical due to leading zeros and occasional non-numeric values.

Text fields were mostly fine, just varied in length and formatting. After cleaning, all columns had consistent data types.

Section 6

Missing Data Analysis

Missing values appear in several columns.

Missing data is mostly concentrated in specific fields. Closure-related fields are often missing because some complaints are still open. Latitude, longitude, and ZIP code are missing in some rows, probably due to incomplete reporting.

Patterns suggest a mix of predictable missing data (like open complaints missing closure dates) and random missingness (like missing coordinates). Columns with significant missing data were flagged, and we documented whether to keep, fill, or exclude them depending on analysis needs.

Section 7

Duplicate Analysis

We looked for both exact and logical duplicates.

Exact duplicates were very few after SQL ingestion. The Unique Key confirmed most records are unique.

Logical duplicates may exist when multiple complaints report the same issue. These were not removed since they reflect real repeated requests rather than errors.

Section 11

Data Consistency and Validity Checks

We ran checks to ensure data is internally consistent.

Borough names were standardized, capitalization issues fixed, and temporal logic verified (closed dates never precede created dates). Records that broke these rules were excluded from response-time calculations.

After cleaning, there were no major inconsistencies or invalid values remaining.

Risks, Limitations, and Assumptions

Limitations include working with a sampled subset instead of the full dataset, missing closure info for open complaints, and reporting bias across boroughs and complaint types.

We made assumptions about handling missing data, keeping outliers, and temporal validity during analysis.
